# Правила работы AI-агентов в LoreForge

Этот файл задает правила для Codex и других AI-агентов, работающих с проектом LoreForge.

## 1. Язык документации

Вся документация проекта должна писаться на русском языке:

- `README.md`;
- `docs/*.md`;
- отчеты;
- комментарии к архитектурным решениям;
- инструкции запуска;
- changelog;
- database workflow;
- dev environment docs.

Код, имена файлов, имена переменных, таблиц, колонок, API routes и технические идентификаторы остаются на английском языке.

## 2. Перед внесением изменений

Перед изменениями агент должен:

- прочитать актуальную структуру проекта;
- проверить существующие документы;
- не дублировать уже существующую информацию;
- не ломать текущую БД и миграции;
- не удалять файлы без явной причины;
- делать маленькие понятные изменения.

## 3. Технологический стек

Использовать современные технологии и лучшие практики:

- TypeScript;
- Next.js / React для frontend;
- Node.js backend, если backend выделяется отдельно;
- PostgreSQL 16;
- Docker Compose;
- Prisma или Drizzle, если будет ORM;
- Tailwind CSS;
- ESLint;
- Prettier;
- Zod для валидации;
- Vitest / Playwright по необходимости;
- pnpm как основной package manager.

Если для качественного результата нужна библиотека или инструмент, агент должен:

- предложить установку;
- объяснить, зачем она нужна;
- добавить ее через package manager проекта;
- обновить документацию;
- не ставить случайные устаревшие пакеты;
- выбирать актуальные стабильные версии;
- проверять совместимость.

## 4. База данных

Правила работы с БД:

- не менять исходную схему без необходимости;
- новые изменения делать миграциями;
- соблюдать порядок применения SQL:
  1. `dnd_campaign_schema.sql`;
  2. `outputs/loreforge_schema_upgrade.sql`;
  3. `outputs/seed_loreforge_demo.sql`, когда появится;
- проверять миграции через PostgreSQL 16;
- сохранять совместимость со старой схемой;
- не удалять старые plugin-таблицы без отдельного решения;
- использовать `updated_at`, `visibility`, индексы и foreign keys аккуратно.

## 5. Документация

Для документации:

- каждый новый workflow документировать в `docs/`;
- если добавлен скрипт, описать, как его запускать;
- если добавлен env-параметр, обновить `.env.example`;
- если добавлена команда, обновить `README.md` или соответствующий документ;
- писать кратко, понятно и по-русски.

## 6. Git

Правила Git:

- не коммитить секреты;
- не коммитить `.env`;
- не коммитить `node_modules`;
- не коммитить временные логи;
- коммиты делать небольшими и осмысленными;
- перед коммитом запускать проверки, если они есть.

## 7. Проверки качества

После изменений запускать доступные проверки:

- PowerShell scripts;
- `docker compose`;
- database check scripts;
- typecheck;
- lint;
- tests, если появятся.

Если проверку невозможно запустить, явно написать почему.

## 8. UI-стиль

Стиль LoreForge: современный dark SaaS dashboard.

Не делать:

- фэнтези-перегруз;
- лишние гримуары;
- карты Таро;
- кубики на фоне;
- старые книги как основной визуальный мотив.

Визуальная идея: Linear / Notion / Vercel + легкий мистический акцент.

Цвета:

- фон: `#0B0F17`;
- карточки: `#111827`;
- границы: `#273244`;
- текст: `#F5F2EA`;
- вторичный текст: `#9CA3AF`;
- акцент: `#8B5CF6`;
- золото: `#D6A84F`;
- опасность: `#B84A4A`.

## 9. Продуктовая модель

LoreForge - не просто Roll20-клон.

Главная идея: кампания как центр мира.

Важные домены:

- Campaign Core;
- Users & Access;
- Characters;
- NPC;
- Locations;
- Scenes;
- Inventory;
- Chat;
- Dice;
- Session Journal;
- GM Notes;
- Tags;
- Investigations;
- Entity Relations;
- World Plugins;
- Plugin Features;
- Campaign Plugins;
- Visibility & Secrets.

Продуктовые правила:

- пользователь не равен персонажу;
- инвентарь принадлежит персонажу, NPC, локации или кампании, но не напрямую пользователю;
- секреты ГМа и `visibility` - ключевая фишка;
- плагины миров не должны быть зашиты в ядро.

## 10. Backend/API

Сначала делать API под реальные экраны:

- campaigns;
- campaign dashboard;
- characters;
- NPC;
- locations;
- chat;
- dice rolls;
- session logs;
- world plugins.

Правила:

- не городить сложную архитектуру раньше времени;
- DTO/validation делать через Zod или аналог;
- ошибки API делать понятными;
- access/visibility service считать отдельным важным слоем.

## 11. Frontend

Сначала делать UI-скелет, затем подключение к API.

Базовые компоненты:

- `AppShell`;
- `Sidebar`;
- `Topbar`;
- `Card`;
- `Button`;
- `Input`;
- `Badge`;
- `ProgressBar`;
- `CampaignCard`;
- `CharacterCard`;
- `NPCDossierCard`;
- `LocationCard`;
- `ChatMessage`;
- `DiceRollPanel`;
- `PluginCard`;
- `SecretBlock`.

Скрытые данные ГМа показывать через `SecretBlock` с замком и пунктирной рамкой.

## 12. Безопасность

Правила безопасности:

- не хранить пароли в открытом виде;
- не выводить секреты в логи;
- не коммитить токены;
- `.env.example` должен содержать только примеры;
- реальные ключи должны быть только в `.env`.

## 13. Формат финального отчета

В конце каждой выполненной задачи агент должен писать:

- что сделано;
- какие файлы изменены;
- какие команды запускались;
- какие проверки прошли;
- что делать дальше.
